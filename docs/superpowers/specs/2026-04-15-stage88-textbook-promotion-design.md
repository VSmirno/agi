# Stage 88 — Knowledge Flow: Textbook Promotion

**Date:** 2026-04-15  
**Status:** Draft  
**Ideological debt:** Principle 5 (Knowledge flow). Verified death hypotheses die with the runtime — they are not passed to the next generation. Each new run rediscovers what the previous run already learned.

---

## Context

After Stage 87, `HypothesisTracker` forms and verifies hypotheses like:
- `zombie + drink < 3.0` [7/13, rate=0.54]
- `dehydration + drink < 3.0` [3/3, rate=1.00]

These hypotheses are in-memory only. A new run starts with no knowledge of them, wastes episodes rediscovering them, and sets conservative thresholds from scratch.

Stage 88 closes this loop: once a hypothesis is stable enough, it is written to `configs/promoted_hypotheses.yaml`. The next generation loads these promoted hypotheses and uses them to:
1. Pre-initialize `PostMortemLearner` thresholds (immediate defensive improvement)
2. Pre-populate `HypothesisTracker` (curiosity skips rediscovery, explores new unknowns)

---

## Decision Log

| Decision | Choice | Rationale |
|---|---|---|
| Generation definition | 1 run = 20 episodes | Matches existing experiment structure |
| Eval structure | 5 sequential runs (gen1…gen5) | Sufficient to observe generational improvement |
| Agent effect | Both: threshold init + tracker init | A alone = immediate survival; B alone = faster curiosity. Together: max gen5/gen1 gap |
| Promotion threshold | `support_rate >= 0.5 AND n_observed >= 5` | Filters weak correlations (zombie+food rate=0.31), accepts reliable ones (zombie+drink rate=0.54) |
| Storage | Separate `configs/promoted_hypotheses.yaml` | Textbook = parental knowledge; promoted = accumulated experience. Different semantics, easy to reset |
| Architecture | Thin `TextbookPromoter` class | Single-purpose, testable, explicit lifecycle in experiment runner |

---

## Architecture

```
configs/promoted_hypotheses.yaml   ← persistent store (separate from crafter_textbook.yaml)
         │                    ▲
         ▼ load               │ save (merge)
  TextbookPromoter  ──────────┘
         │
    ┌────┴──────────────────────────┐
    ▼                               ▼
HypothesisTracker(initial=promoted)  PostMortemLearner.from_promoted(promoted)
    │                               │
    └────────── experiment runner ──┘
                  (gen1 → gen5)
```

## Design

### Block 1 — `TextbookPromoter`

**File:** `src/snks/agent/textbook_promoter.py`

```python
PROMOTE_SUPPORT_RATE = 0.5
PROMOTE_MIN_OBSERVED = 5

class TextbookPromoter:
    def should_promote(self, h: DeathHypothesis) -> bool:
        return h.support_rate >= PROMOTE_SUPPORT_RATE and h.n_observed >= PROMOTE_MIN_OBSERVED

    def load(self, path: Path) -> list[DeathHypothesis]:
        """Load promoted hypotheses from YAML. Returns [] if file absent or empty."""

    def save(self, hypotheses: list[DeathHypothesis], path: Path) -> None:
        """Write promotable hypotheses to YAML.
        Merge logic: if hypothesis (cause, vital) already exists, keep the one
        with higher n_observed (richer observation wins).
        """
```

**YAML format** (`configs/promoted_hypotheses.yaml`):
```yaml
# Auto-generated by TextbookPromoter. Do not edit manually.
# Promoted when: support_rate >= 0.5 AND n_observed >= 5
hypotheses: []
```

After gen1, example populated file:
```yaml
hypotheses:
  - cause: zombie
    vital: drink
    threshold: 3.0
    n_supporting: 7
    n_observed: 13
  - cause: dehydration
    vital: drink
    threshold: 3.0
    n_supporting: 3
    n_observed: 3
```

Required fields: `cause`, `vital`, `threshold`, `n_supporting`, `n_observed`. These are sufficient to reconstruct `support_rate` and `is_verifiable` without storing derived values.

### Block 2 — `HypothesisTracker` initialization

**File:** `src/snks/agent/death_hypothesis.py`

Add optional `initial` parameter to `HypothesisTracker.__init__` and a separate `_promoted` store:

```python
def __init__(self, initial: list[DeathHypothesis] | None = None) -> None:
    self._records: list[dict] = []
    self._hypotheses: list[DeathHypothesis] = []
    self._promoted: list[DeathHypothesis] = list(initial or [])
```

**Why `_promoted` instead of injecting into `_hypotheses` directly:**

`_rebuild_hypotheses()` derives `_hypotheses` exclusively from `_records`. If promoted hypotheses were injected into `_hypotheses` only, the first call to `record()` would overwrite them entirely — silently erasing all generation knowledge after one death. This is the critical bug avoided by using a separate `_promoted` list.

**Merge in `_rebuild_hypotheses()`:**

At the end of `_rebuild_hypotheses()`, merge `_promoted` into the freshly-derived list:

```python
def _rebuild_hypotheses(self) -> None:
    # ... derive hypotheses from _records as before ...

    # Merge promoted: if (cause, vital) not yet observed in this run,
    # carry forward the promoted entry so it remains active and verifiable.
    # If the current run has more observations, the live entry wins.
    live_keys = {(h.cause, h.vital) for h in hypotheses}
    for ph in self._promoted:
        key = (ph.cause, ph.vital)
        if key not in live_keys:
            hypotheses.append(ph)
        # else: live run has data — it will overwrite the promoted entry
        # once n_observed exceeds the promoted n_observed, preventing stale priors

    self._hypotheses = hypotheses
```

**Effect:**
- `active_hypothesis()` can immediately return a promoted hypothesis (from gen N-1)
- Once the current gen accumulates enough deaths from the same cause, the live-derived entry replaces the promoted one
- `is_verifiable` is satisfied immediately for promoted entries (they already have n_observed ≥ 5, n_supporting ≥ 3)
- Curiosity is directed toward `(cause, vital)` pairs NOT yet in `_promoted`

### Block 3 — `PostMortemLearner.from_promoted`

**File:** `src/snks/agent/post_mortem.py`

```python
@classmethod
def from_promoted(cls, hypotheses: list[DeathHypothesis], lr: float = 0.1) -> PostMortemLearner:
    """Create a PostMortemLearner with thresholds pre-initialized from promoted hypotheses."""
    learner = cls(lr=lr)
    for h in hypotheses:
        bump = min(h.support_rate * 2.0, 2.0)  # max +2.0
        if h.vital == "drink":
            learner.drink_threshold = cls._clamp(
                max(learner.drink_threshold, 3.0 + bump), _THRESHOLD_MIN, _THRESHOLD_MAX
            )
        elif h.vital == "food":
            learner.food_threshold = cls._clamp(
                max(learner.food_threshold, 3.0 + bump), _THRESHOLD_MIN, _THRESHOLD_MAX
            )
        if h.cause in ("zombie", "skeleton"):
            learner.health_weight = cls._clamp(
                max(learner.health_weight, 1.0 + h.support_rate), _WEIGHT_MIN, _WEIGHT_MAX
            )
    return learner
```

**Formula rationale:** `3.0 + support_rate * 2.0` is a deliberate conservative overestimate from the default base (3.0). It does not try to restore the exact per-generation convergence value because we don't store it. Instead it computes a fresh prior from the correlation strength: `rate=0.5` → threshold raised by 1.0 (moderate), `rate=1.0` → raised by 2.0 (maximum). This is intentionally generous — the threshold will be pulled back down by `update()` within the new generation if it overcorrects.

**Example** — after gen1 with `zombie+drink rate=0.54, dehydration+drink rate=1.00` (processed in this order):
- After zombie+drink: `drink_threshold = cls._clamp(max(3.0, 3.0+1.08), ...) = 4.08`
- After dehydration+drink: `drink_threshold = cls._clamp(max(4.08, 3.0+2.0), ...) = 5.0`
- After zombie (cause): `health_weight = cls._clamp(max(1.0, 1.0+0.54), ...) = 1.54`
- After dehydration (not zombie/skeleton): health_weight unchanged
- Final: `drink_threshold=5.0, health_weight=1.54`

Note: test must pass hypotheses in a deterministic order (zombie+drink before dehydration+drink) to match expected intermediate values.

### Block 5 — `experiments/exp108_stage88.py`

5-generation loop. Each generation:
1. Load promoted hypotheses from file
2. Init `PostMortemLearner.from_promoted(promoted)`
3. Init `HypothesisTracker(initial=promoted)`
4. Run 20 episodes
5. Filter `tracker.all_hypotheses()` by `should_promote()`
6. Save (merge) to `promoted_hypotheses.yaml`
7. Log gen results

```python
N_GENS = 5
N_EPISODES = 20
promoter = TextbookPromoter()
promoted_path = Path("configs/promoted_hypotheses.yaml")
results = {}

for gen in range(1, N_GENS + 1):
    promoted = promoter.load(promoted_path)
    learner = PostMortemLearner.from_promoted(promoted)
    tracker = HypothesisTracker(initial=promoted)

    gen_survivals = run_episodes(N_EPISODES, learner=learner, tracker=tracker)
    results[gen] = {
        "avg_survival": mean(gen_survivals),
        "n_promoted_cumulative": len(promoted),
    }

    to_promote = [h for h in tracker.all_hypotheses() if promoter.should_promote(h)]
    promoter.save(to_promote, promoted_path)
    log(f"gen{gen}: avg_survival={results[gen]['avg_survival']:.1f}, "
        f"promoted_in={len(promoted)}, promoted_out={len(to_promote)}")
```

## Data Flow (one generation)

```
START gen N
  │
  ├─ load promoted_hypotheses.yaml  → list[DeathHypothesis]
  │
  ├─ PostMortemLearner.from_promoted(promoted)
  │    zombie+drink rate=0.54 → drink_threshold = 4.08
  │    dehydration+drink rate=1.0 → drink_threshold = max(4.08, 5.0) = 5.0
  │
  ├─ HypothesisTracker(initial=promoted)
  │    promoted already in _hypotheses → curiosity seeks NEW unknowns
  │
  └─ run_episodes(20)
       per episode:
         record(attribution, vitals_at_death)
         active_hypothesis() → CuriosityStimulus.death_relevance()

END gen N
  │
  ├─ filter by should_promote() → merge save
  └─ log avg_survival
```

## Gates

**Baseline:** gen1 starts with empty `promoted_hypotheses.yaml` — equivalent to a standard Stage 87 run (expected avg_survival ≈ 187 based on Stage 87 data).

| Gate | Condition | Type |
|---|---|---|
| Primary | `gen5_avg_survival / gen1_avg_survival >= 1.20` OR `gen5_avg_survival >= 210` | Main gate |
| Secondary | `n_promoted >= 2` after gen2 | Sanity: promotion actually fires |

The OR clause handles the case where gen1 accidentally performs unusually well (>190), making the ratio gate unachievable. A standalone survival floor of 210 (≈ +12% over Stage 87 baseline) suffices to declare the mechanism works.

## Testing

**`tests/test_textbook_promoter.py`** — unit tests:

- `should_promote`: rate=0.49 → False, rate=0.50 → True, n_observed=4 → False, n_observed=5 → True
- `save` + `load` round-trip: 2 hypotheses saved → loaded identically
- `save` merge: existing n_observed=10, new n_observed=7 → existing wins (n_observed=10 kept)
- `from_promoted`: pass `[zombie+drink rate=0.54, dehydration+drink rate=1.0]` in that order → `drink_threshold = 5.0`, `health_weight = 1.54`
- `HypothesisTracker(initial=promoted)`: promoted hypotheses immediately in `all_hypotheses()`, `n_verifiable() >= len(promoted)`
- `HypothesisTracker` after first `record()` call: promoted hypotheses for unseen `(cause, vital)` pairs remain in `all_hypotheses()` (merge logic preserves them)

**No local integration tests** — gen-loop is validated by the experiment on minipc.
