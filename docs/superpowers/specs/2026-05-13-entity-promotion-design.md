# Entity Promotion — First Iteration of TextbookPromoter (PCCS step 2.0)

**Date:** 2026-05-13
**Status:** Design (pre-implementation)
**Companion docs:** [`docs/IDEOLOGY.md`](../../IDEOLOGY.md), [`docs/architecture-report-2026-05-11.md`](../../architecture-report-2026-05-11.md), [`docs/superpowers/specs/2026-05-12-outcome-role-design.md`](2026-05-12-outcome-role-design.md)

---

## 1. Problem this closes

Ideology principle 5 — *Knowledge flow (textbook ← experience)* — is the
last red ❌ in the architecture scorecard (§7 of the architecture report).
`TextbookPromoter` exists structurally (Stage 88) but
`configs/promoted_hypotheses.yaml` has stayed empty: nothing actually
promotes durable experience back into the declarative layer. The agent
forgets what it learned about its own environment between episodes.

A separate but related gap (architecture-report §"Новый gap" appended
2026-05-12) is that `CrafterSpatialMap` is fully episode-local — every
`reset()` wipes it. Tables and furnaces the agent placed in episode N
have to be discovered all over again in episode N+1 even though the
underlying world (per seed) is the same.

Both gaps share one fix: an experience→facts loop that promotes
*persistent entity observations* (placed-objects, in this iteration) so
the next episode of the same seed starts already knowing where its
existing tables sit.

## 2. The ontological frame

The full world model rests on five primitives — **Entity / Relation /
Action / Context / Goal** — agreed in brainstorming on 2026-05-13.
This iteration promotes only **Entity**-typed knowledge nodes. Later
iterations add Relation, Action, Context, Goal node types into the same
YAML schema without changing the existing entries.

An Entity node carries:
- a stable identifier (concept + position for placed-objects)
- a type tag (`table`, `furnace`, future: `cow_spawn_zone`, `water_pool`)
- provenance: source = `entity_observation`, evidence = how often it was
  observed, in how many episodes

The promoter is therefore a **knowledge compiler** even in this minimal
form — it reads `CrafterSpatialMap._map` (substrate-like accumulator),
detects stable Entity occurrences, and writes typed knowledge nodes.
The same compiler will later read SDM outcome traces and emit
Action-precondition and Relation nodes.

## 3. Storage and scope

**Per-seed YAML store**: `_docs/wm/seed{N}_promoted.yaml`. Sits next to
the existing `world_model_seed{N}.pt` binary. Reasons:

- Crafter generates a fresh world per seed. Position `(28, 33)` in
  seed 17 is *not* the same tile as in seed 27. Promoted Entity
  positions are only valid within their seed lineage.
- Human-readable text format. The binary `.pt` stays for SDM tensors;
  declarative knowledge stays in YAML — same separation as
  `crafter_textbook.yaml` (declarative) vs SDM content (statistical).
- Matches per-seed lineage already adopted for outcome-role
  persistence (`_docs/wm/seed{N}.pt`).
- Gitignored alongside the other `_docs/wm/*` files.

Schema (one extensible YAML file per seed):

```yaml
# Auto-generated by TextbookPromoter (entity iteration). Do not edit manually.
schema_version: 1
nodes:
  - type: entity_observation
    body:
      concept: table
      position: [28, 33]
    provenance:
      source: spatial_map_compiler
      observation_count: 4          # times seen across all episodes
      observed_in_episodes: 2       # number of distinct episodes
      first_seen_episode: 0
      last_seen_episode: 1
      confidence: 1.0
  - type: entity_observation
    body:
      concept: furnace
      position: [29, 41]
    provenance: { ... }
```

`schema_version: 1` is the wedge for future Relation / Action /
Context / Goal node types — they get added under `nodes:` with
`type: relation_observation` etc.

## 4. Promotion rule (this iteration)

At episode end, walk `spatial_map._map`:

- For each `(y, x) → (concept, conf, count)` where:
  - `concept ∈ PROMOTABLE_ENTITIES` (initially `{"table", "furnace"}`)
  - `count >= 2` (seen in at least two perception cycles this episode —
    permissive but rejects single-frame mis-classifications)
- Emit/merge a node `entity_observation` for that `(concept, position)`.

Merge semantics:
- Key on `(type, body.concept, body.position)`.
- New observation increments `observation_count` and updates
  `last_seen_episode`.
- If an earlier promoted node is *not* re-observed in the current
  episode, leave it alone — no decay on first absence. Tables persist
  unless explicitly destroyed (Crafter's `place_stone` over a tile
  could overwrite it, but that's a separate concern handled when the
  spatial map sees the new tile content).

The threshold `observation_count >= 2` is deliberately permissive
because placed-objects are stable — once observed reliably they don't
need long-running statistical support. If a later iteration wants
stricter (e.g. for transient entities like cows), `PROMOTABLE_ENTITIES`
gets per-concept thresholds in the same code path.

## 5. Load-time integration

At the start of `run_vector_mpc_episode` (after `model.load` for the
outcome-role substrate but before the first perception step), if the
promoted-YAML exists for the current seed, the promoter loads each
`entity_observation` node and pre-fills `spatial_map._map[(y, x)] =
(concept, confidence, observation_count)`.

Effect:
- `CrafterSpatialMap.find_nearest("table", player_pos)` already returns
  the placed table at episode start, even before the agent walks into
  range to observe it directly.
- The planner's `generate_candidate_plans` adjacency-check (via
  `near_requirements`) sees the existing table and *does not* generate
  a redundant `[place_table → make_X]` chain plan. Single make plan
  routes through `expand_to_primitive`'s navigation branch and steps
  the agent toward the known table.

This is exactly the failure mode that left 3 stacked tables on
seed 17 ep 0 during the Variant B work (architecture-report §
"placed-object override in spatial_map"). That fix was a write-side
guard; entity promotion is the read-side counterpart that makes the
fix work across episodes.

## 6. What this design deliberately does NOT do

- **No Relation / Action / Context / Goal nodes**. Those land in
  subsequent iterations. The YAML schema is open to them but the code
  in this iteration only emits Entity nodes.
- **No statistical pattern detection over SDM**. Stable `(c, a)` →
  outcome regularities live in `model.memory` already; promoting them
  is a future iteration that compiles SDM patterns into Action nodes.
- **No de-promotion on contradiction**. If a future episode shows a
  promoted table is gone (the tile reads as something else with high
  confidence), this iteration doesn't drop the node. That belongs in
  an explicit "invalidation" mechanism alongside Relation promotion.
- **No threshold-tuning across episodes**. Hardcoded
  `observation_count >= 2`. Adaptive thresholds belong in a later
  meta-promoter that tunes its own parameters.
- **No cross-seed knowledge transfer**. Each seed has its own file;
  no global pool. (Crafter's per-seed worlds make this moot.)

## 7. Risks and checks

| Risk | Detection | Mitigation |
|---|---|---|
| Stale entity nodes after world change (table destroyed) | grep promoted YAML after a `place_stone` over a table | First iteration accepts this. Documented as deferred — invalidation comes with Relation promotion. |
| YAML grows unbounded across many episodes | `wc -l` the seed file | Bounded by world size: max ~50 placed-objects per seed in practice. Each entry ~10 lines. Real cap ~500 LOC for the seed's lifetime. |
| Load-time pre-fill corrupts spatial_map | Compare `find_nearest("table")` before/after with no other writes | Pre-fill calls `spatial_map.update(pos, concept, conf=1.0)` which respects the placed-object override added in Variant B. Other Bug-B guards still apply. |
| Stage 91 determinism breaks | Two-run byte-compare under strict eval after load | Loading is deterministic given the same YAML. No new RNG draws. Verified in step 6 of the implementation plan. |
| Promoter promotes spurious player-tile observations | Compare promoted nodes to actual placed-object actions in the trace | Promotion gate `concept ∈ PROMOTABLE_ENTITIES` excludes `player` and all natural-world tiles. Player-tile mis-classifications never produce promotable entries. |

## 9. Implementation plan

1. **TextbookPromoter v2 + unit tests.** Schema_version=1 YAML.
   `collect_entity_observations`. `merge_nodes` with key on
   `(type, body.concept, body.position)`. Roundtrip tests. ~1 hr.
2. **vector_mpc_agent integration.** Load promoted YAML at episode
   start (gated by `enable_outcome_learning` — same kwarg, no new
   plumbing); call promoter at episode end. ~30 min.
3. **Local smoke test on a 30-step fake env episode** with two placed
   tables to validate pre-fill and emit paths. ~15 min.
4. **HyperPC validation.** Seed 17 gen1 (no promoted file) → gen2
   (file present). Expected: gen2's first `place_table` happens
   later or not at all if gen1's table positions cover the planning
   need. Compare with the Variant B / outcome-role gen2 baseline.
   ~10 min wall.
5. **5-seed multiseed.** Aggregate gen1 vs gen2 placed-table count
   and time-to-first-craft. Expected: gen2 *fewer* place_table
   actions on average, *faster* first make_sword. ~30 min.
6. **Determinism check.** Two runs same seed both with identical
   loaded promoted-YAML → byte-identical action+position traces.
   ~10 min.
7. **Architecture report + README update.** Lift principle 5
   from ❌ to 🟢. Tick the "persistent placed-object memory" gap
   from §"Новый gap". Note that subsequent promoter iterations will
   add Relation / Action / Context / Goal nodes into the same
   `seed{N}_promoted.yaml` schema. ~15 min.

Total target: ~2.5–3 hours of focused work plus ~1 hour HyperPC wall.

Maintainer agrees that:

  builds the schema other iterations extend).

- The five-primitive ontology (Entity / Relation / Action / Context /
